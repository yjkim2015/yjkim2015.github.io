> **한 줄 요약**: Next.js는 React 기반 풀스택 프레임워크로, SSG·SSR·ISR·CSR 렌더링 전략을 상황에 맞게 조합하고 App Router의 서버 컴포넌트로 성능과 SEO를 동시에 잡을 수 있습니다.

---

## 비유로 이해하기

음식 제공 방식에 비유하면 렌더링 전략을 쉽게 이해할 수 있습니다. **CSR(Client-Side Rendering)**은 일반 식당입니다. 손님이 오면 그때서야 요리를 시작하고, 다 완성되기 전까지 손님은 빈 접시를 봐야 합니다. **SSG(Static Site Generation)**는 뷔페입니다. 미리 음식을 만들어 진열해두고, 손님이 오면 즉시 제공합니다. 메뉴가 바뀌려면 다음 영업까지 기다려야 합니다. **ISR(Incremental Static Regeneration)**은 정기배송 도시락입니다. 미리 만들어두되, 주기적으로 새로 만들어 교체합니다. **SSR(Server-Side Rendering)**은 개인 맞춤 즉석 조리입니다. 손님 정보에 맞게 그 자리에서 만들어 제공합니다.

Next.js는 이 네 가지 방식을 **하나의 프레임워크 안에서 페이지별로 선택**할 수 있게 해줍니다. 블로그 게시글은 SSG로, 개인화된 대시보드는 SSR로, 실시간 채팅은 CSR로 구현하면 됩니다.

App Router(Next.js 13+)는 여기서 한 단계 더 나아가, 서버에서만 실행되는 Server Component와 브라우저에서 실행되는 Client Component를 세밀하게 나누어 JavaScript 번들 크기를 최소화합니다.

| 전략 | 빌드 시 생성 | 요청 시 생성 | SEO | 실시간 데이터 | 사용 예 |
|------|------------|------------|-----|-------------|---------|
| CSR | X | O (클라이언트) | 나쁨 | 가능 | 대시보드, 관리자 페이지 |
| SSG | O | X | 좋음 | 불가 | 블로그, 문서, 마케팅 |
| ISR | O | X (주기 갱신) | 좋음 | 준실시간 | 뉴스, 상품 목록 |
| SSR | X | O (서버) | 좋음 | 가능 | 개인화 피드, 실시간 |

#### 왜 이게 중요한가?

렌더링 전략 선택은 사용자 경험과 SEO에 직결됩니다. 블로그 글을 CSR로 구현하면 검색 엔진이 빈 HTML만 보게 되어 SEO가 망가집니다. 반대로 개인화된 대시보드를 SSR로 구현하면 서버 부하가 크고 응답이 느려집니다. 각 전략의 트레이드오프를 이해하고 페이지 성격에 맞게 선택해야 합니다.

## 3. Server Components vs Client Components

```mermaid
flowchart TD

    end

    subgraph "Client Component ('use client')"
        CC["브라우저에서 실행\nJS 번들에 포함\nuseState, useEffect 사용 가능\n이벤트 핸들러 가능\nwindow, document 접근 가능"]
    end

    subgraph "조합 패턴"
        SP["Server Page\n(데이터 페칭)"]
        CP["Client Button\n(인터랙션)"]
    end

    SC -->|"Client Component\n자식으로 포함 가능"| CC
    SP -->|"props로 데이터 전달"| CP

    style CC fill:#f39c12,color:#fff
```

```tsx
// ① Server Component: DB 직접 접근, 번들 크기 0
// app/products/[id]/page.tsx
async function getProduct(id: string) {
    const product = await db.product.findUnique({ where: { id } });
    return product;
}

export default async function ProductPage({ params }: { params: { id: string } }) {
    const product = await getProduct(params.id);

    return (
        <div>
            <h1>{product.name}</h1>
            <p>{product.description}</p>
            <p>₩{product.price.toLocaleString()}</p>
            {/* Client Component를 자식으로 포함 */}
            <AddToCartButton productId={product.id} />
        </div>
    );
}

// SEO 메타데이터 - 서버에서 동적 생성
export async function generateMetadata({ params }) {
    const product = await getProduct(params.id);
    return {
        title: product.name,
        description: product.description,
        openGraph: { images: [product.imageUrl] }
    };
}

// ② Client Component: 상호작용이 필요한 부분만 분리
// components/AddToCartButton.tsx
'use client'

import { useState } from 'react';
import { useCartStore } from '@/store/cart';

export function AddToCartButton({ productId }: { productId: string }) {
    const [added, setAdded] = useState(false);
    const addToCart = useCartStore(state => state.addItem);

    const handleClick = async () => {
        await addToCart(productId);
        setAdded(true);
        setTimeout(() => setAdded(false), 2000);
    };

    return (
        <button onClick={handleClick} disabled={added}>
            {added ? '담겼습니다!' : '장바구니 담기'}
        </button>
    );
}
```

#### Server Component vs Client Component 선택 기준

| 상황 | 선택 | 이유 |
|------|------|------|
| 데이터 페칭 (DB, API) | Server | JS 번들에 포함 안 됨 |
| useState, useEffect 필요 | Client | Hook은 브라우저에서만 동작 |
| 이벤트 핸들러 (onClick 등) | Client | 브라우저 이벤트 |
| window, document 접근 | Client | 브라우저 API |
| 무거운 라이브러리 (그래프 등) | 가능하면 Server | 번들 크기 절감 |
| 민감한 API key 사용 | Server | 클라이언트 노출 방지 |

## 4. API Routes

```typescript
// app/api/orders/route.ts
import { NextRequest, NextResponse } from 'next/server';

export async function GET(request: NextRequest) {
    const searchParams = request.nextUrl.searchParams;
    const userId = searchParams.get('userId');

    if (!userId) {
        return NextResponse.json({ error: 'userId required' }, { status: 400 });
    }

    const orders = await db.order.findMany({
        where: { customerId: userId },
        include: { items: true }
    });

    return NextResponse.json(orders);
}

export async function POST(request: NextRequest) {
    const body = await request.json();

    // 서버 세션으로 인증 확인 (토큰 클라이언트 노출 없음)
    const session = await getServerSession();
    if (!session) {
        return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
    }

    const order = await db.order.create({
        data: { ...body, customerId: session.user.id }
    });
    return NextResponse.json(order, { status: 201 });
}

// app/api/orders/[id]/route.ts - 동적 세그먼트
export async function GET(
    request: NextRequest,
    { params }: { params: { id: string } }
) {
    const order = await db.order.findUnique({ where: { id: params.id } });
    if (!order) {
        return NextResponse.json({ error: 'Not Found' }, { status: 404 });
    }
    return NextResponse.json(order);
}

export async function PATCH(
    request: NextRequest,
    { params }: { params: { id: string } }
) {
    const body = await request.json();
    const updated = await db.order.update({
        where: { id: params.id },
        data: body
    });
    return NextResponse.json(updated);
}
```

## 5. 미들웨어 (Middleware)

미들웨어는 요청이 페이지에 도달하기 **전에 실행**됩니다. Edge Runtime에서 실행되어 매우 빠릅니다.

```mermaid
sequenceDiagram
    participant Browser as 브라우저
    participant Edge as Edge Runtime
    participant MW as middleware.ts
    participant Page as 페이지/API

    Browser->>Edge: GET /dashboard
    Edge->>MW: 미들웨어 실행
    MW->>MW: 쿠키에서 auth-token 확인
    alt 토큰 없음
        MW-->>Browser: 302 Redirect /login?redirect=/dashboard
    else 토큰 있음
        MW->>Page: 요청 통과 + 헤더 추가
        Page-->>Browser: 200 페이지 응답
    end
```

```typescript
// middleware.ts (프로젝트 루트에 위치)
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
    const token = request.cookies.get('auth-token')?.value;
    const { pathname } = request.nextUrl;

    // 보호된 경로 목록
    const protectedPaths = ['/dashboard', '/orders', '/profile'];
    const isProtected = protectedPaths.some(p => pathname.startsWith(p));

    if (isProtected && !token) {
        const loginUrl = new URL('/login', request.url);
        loginUrl.searchParams.set('redirect', pathname);
        return NextResponse.redirect(loginUrl);
    }

    // 이미 로그인한 상태에서 로그인 페이지 접근
    if (pathname === '/login' && token) {
        return NextResponse.redirect(new URL('/dashboard', request.url));
    }

    // 모든 요청에 요청 ID 추가 (로깅/추적용)
    const response = NextResponse.next();
    response.headers.set('X-Request-Id', crypto.randomUUID());
    return response;
}

// 미들웨어 적용 경로 설정 (정적 파일 제외)
export const config = {
    matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'],
};
```

### Suspense와 Streaming

> **비유:** Streaming SSR은 레스토랑의 코스 요리와 같습니다. 모든 요리가 완성될 때까지 기다리지 않고, 전채부터 차례로 내보냅니다. 손님(브라우저)은 첫 요리(레이아웃)를 바로 먹기 시작하고, 메인 요리(데이터가 많은 컴포넌트)는 준비되는 대로 제공됩니다.

```mermaid
sequenceDiagram
    participant Browser as 브라우저
    participant Server as Next.js 서버

    Browser->>Server: GET /products
    Server-->>Browser: HTML 스트리밍 시작
    Note over Browser: 레이아웃 즉시 표시
    Server-->>Browser: ProductList 청크 완료
    Note over Browser: 상품 목록 표시
    Server-->>Browser: Recommendations 청크 완료
    Note over Browser: 추천 상품 표시
    Note over Browser: 전체 페이지 완성
```

```tsx
// Suspense로 느린 컴포넌트 분리 → 나머지 먼저 표시
import { Suspense } from 'react';

export default async function ProductsPage() {
    return (
        <div>
            <h1>상품 목록</h1>
            {/* 빠른 부분: 즉시 스트리밍 */}
            <ProductList />

            {/* 느린 부분: 로딩 UI 먼저, 완료 후 교체 */}
            <Suspense fallback={<RecommendationSkeleton />}>
                <Recommendations />
            </Suspense>
        </div>
    );
}

// 느린 Server Component — DB 쿼리에 3초 소요
async function Recommendations() {
    const items = await db.recommendation.findMany(); // 3초
    return <RecommendationGrid items={items} />;
}
```

### 캐싱 전략

Next.js App Router는 4단계 캐시 레이어를 제공합니다.

| 캐시 레이어 | 위치 | 대상 | 무효화 방법 |
|------------|------|------|------------|
| Request Memoization | 서버 (요청 내) | 같은 fetch 중복 제거 | 요청 종료 시 자동 |
| Data Cache | 서버 (영속) | fetch 응답 | `revalidateTag()`, `revalidatePath()` |
| Full Route Cache | 서버 (영속) | 정적 렌더링 HTML | 재배포, `revalidatePath()` |
| Router Cache | 클라이언트 (세션) | RSC Payload | `router.refresh()`, 자동 만료 |

```tsx
// Data Cache 제어 예시
// 1) 영구 캐시 (SSG와 동일)
fetch(url);  // 기본값: force-cache

// 2) ISR: 60초마다 백그라운드 재검증
fetch(url, { next: { revalidate: 60 } });

// 3) SSR: 매 요청마다 새로 페칭
fetch(url, { cache: 'no-store' });

// 4) 태그 기반 온디맨드 재검증
fetch(url, { next: { tags: ['products'] } });
// Server Action에서:
import { revalidateTag } from 'next/cache';
revalidateTag('products');  // 'products' 태그 캐시 즉시 무효화
```

### 시나리오 1: Server Component에서 민감 정보가 클라이언트에 노출

Server Component에서 DB 비밀번호가 포함된 환경 변수를 로깅하고, 그 결과를 `props`로 Client Component에 전달했다. Server Component 자체는 서버에서만 실행되지만, `props`로 전달된 데이터는 RSC Payload에 직렬화되어 브라우저에 전송된다. 개발자 도구 Network 탭에서 DB 비밀번호가 그대로 노출된다. Server Component에서 Client Component로 전달하는 `props`는 반드시 필요한 데이터만, 민감 정보를 제거한 후 전달해야 한다.

### 시나리오 2: 무한 재검증 루프로 서버 과부하

ISR `revalidate: 1`로 설정한 상품 페이지가 1,000개 있다. 매초 1,000개 페이지를 재생성하면서 서버 CPU가 100%에 도달한다. CDN 캐시 히트율이 급락하고 원본 서버로 요청이 집중되어 전체 사이트가 다운된다. `revalidate` 값은 데이터 변경 빈도에 맞게 설정하고(상품 페이지라면 60~300초), 온디맨드 재검증(`revalidateTag`)으로 실제 변경 시에만 갱신하는 것이 올바르다.

### 시나리오 3: 'use client' 경계 설정 실수로 번들 크기 폭발

최상위 `layout.tsx`에 `'use client'`를 붙였다. 모든 하위 컴포넌트가 Client Component로 취급되어, 서버에서만 실행되어야 할 데이터 페칭 로직, DB 쿼리, 무거운 라이브러리가 모두 클라이언트 JS 번들에 포함된다. 초기 로딩 시간이 5초 이상으로 증가하고 Core Web Vitals가 전부 빨간색이 된다. `'use client'`는 상호작용이 필요한 **말단 컴포넌트**에만 붙여야 한다.

### 시나리오 4: 미들웨어에서 무거운 연산 실행으로 전체 응답 지연

미들웨어에서 JWT 검증을 위해 외부 API를 호출하고, 응답을 파싱한 후 DB 조회까지 수행한다. 미들웨어는 **모든 요청**에 실행되므로, 정적 파일 요청(이미지, CSS, JS)까지 200ms의 지연이 추가된다. 미들웨어는 Edge Runtime에서 실행되므로 경량 연산(쿠키 확인, 리다이렉트)만 수행하고, 무거운 로직은 Server Component나 API Route로 옮겨야 한다.

## 8. 핵심 포인트 정리

| 개념 | 핵심 | 실무 주의점 |
|------|------|------------|
| SSG | 빌드 시 HTML 생성, CDN 배포, 가장 빠름 | 데이터 변경 시 재빌드 필요 |
| ISR | SSG + 주기적 백그라운드 재생성 | `revalidate` 값을 너무 낮게 설정하면 서버 과부하 |
| SSR | 매 요청마다 서버에서 HTML 생성 | TTFB가 느려질 수 있음, 캐싱 전략 필수 |
| CSR | 브라우저에서 JS로 렌더링 | SEO 불가, 초기 로딩 느림 |
| Server Component | 서버에서만 실행, JS 번들 0 | `useState`/`useEffect` 사용 불가 |
| Client Component | 브라우저에서 실행, JS 번들 포함 | 최소 범위로 분리해야 번들 크기 절감 |
| Middleware | 모든 요청 전 실행, Edge Runtime | 경량 연산만, 무거운 로직은 API Route로 |
| `fetch` 캐시 | `force-cache`(기본) → `revalidate` → `no-store` | 태그 기반 온디맨드 재검증 권장 |
| `next/image` | 자동 WebP 변환, lazy loading, CLS 방지 | LCP 이미지에는 `priority` 필수 |
| Dynamic Import | 필요할 때만 컴포넌트 로드 | 브라우저 전용 라이브러리는 `ssr: false` |

## 9. 면접 포인트

**Q1. SSR과 SSG의 차이를 설명하고 각각 언제 사용하는가?**

SSG는 빌드 시 HTML을 미리 생성하여 CDN에 배포하므로 응답이 가장 빠르다. 블로그, 문서, 마케팅 페이지처럼 내용이 자주 바뀌지 않는 페이지에 적합하다. SSR은 매 요청마다 서버에서 HTML을 생성하므로 항상 최신 데이터를 보여줄 수 있다. 로그인 사용자의 개인화 피드, 실시간 데이터가 필요한 페이지에 적합하다. ISR은 두 방식의 장점을 결합하여 정적 페이지를 주기적으로 재생성한다.

**Q2. Server Component와 Client Component의 차이는?**

Server Component는 서버에서만 실행되어 JS 번들에 포함되지 않는다. DB 직접 접근, API Key 사용이 가능하지만 `useState`, `useEffect` 등 React Hook을 사용할 수 없다. Client Component는 `'use client'` 지시어를 붙이며, 브라우저에서 실행되어 사용자 상호작용(이벤트 핸들러, 상태 관리)을 처리한다. 성능을 위해 `'use client'`는 상호작용이 필요한 말단 컴포넌트에만 붙여야 한다.

**Q3. Next.js의 캐싱 레이어를 설명하라.**

4단계다. (1) Request Memoization: 같은 요청 내 중복 fetch를 자동 제거한다. (2) Data Cache: fetch 응답을 서버에 영속 캐싱하며 `revalidateTag`로 무효화한다. (3) Full Route Cache: 정적 렌더링된 HTML/RSC Payload를 캐싱한다. (4) Router Cache: 클라이언트에서 방문한 경로의 RSC Payload를 세션 내 캐싱한다.

**Q4. Middleware는 어디서 실행되고 무엇을 해야 하는가?**

Edge Runtime에서 실행되며 모든 요청 전에 호출된다. 인증 토큰 확인 후 리다이렉트, 지역별 라우팅, A/B 테스트 분기 같은 경량 연산에 적합하다. DB 조회, 외부 API 호출 같은 무거운 작업은 응답 지연을 유발하므로 Server Component나 API Route에서 처리해야 한다.
