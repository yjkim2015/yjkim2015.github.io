> **한 줄 요약**: 레이어드 아키텍처는 소프트웨어를 책임에 따라 수평 계층으로 나누는 가장 보편적인 구조로, 대부분의 Spring 애플리케이션이 이 패턴을 따른다.

## 비유로 시작하기

패스트푸드 음식점을 생각해보세요.

- **카운터(Presentation)**: 손님과 대화하고 주문을 받음
- **주방(Business Logic)**: 실제 음식을 만드는 핵심 작업
- **냉장고/창고(Persistence)**: 재료를 보관하고 꺼냄

손님은 주방과 직접 대화하지 않고, 주방은 손님의 얼굴을 볼 필요가 없습니다. 카운터가 바뀌어도 주방 레시피는 변하지 않습니다. 각 역할이 명확하게 분리되어 있습니다.

레이어드 아키텍처(Layered Architecture)는 소프트웨어를 이처럼 **책임에 따라 수평적으로 나눈 구조**입니다. 가장 오래되고 가장 널리 사용되는 아키텍처 패턴이며, 대부분의 Spring 애플리케이션이 이 구조를 따릅니다.

---

## 왜 레이어드 아키텍처인가?

```
레이어드가 없으면:
  Controller 하나가 HTTP 파싱 + 비즈니스 로직 + DB 쿼리를 모두 처리
  → 코드 수천 줄짜리 "God Class" 탄생
  → 한 줄 수정이 전혀 다른 기능을 망가뜨림
  → 단위 테스트 불가 (모든 것이 뒤엉킴)

레이어드 아키텍처로:
  Controller: HTTP만 담당
  Service: 비즈니스 로직만 담당
  Repository: DB만 담당
  → 각 레이어를 독립적으로 수정, 테스트 가능
```

## 각 계층 역할

### Presentation Layer (표현 계층)

사용자나 클라이언트와의 **입출력을 담당**합니다.

```java
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @PostMapping
    public ResponseEntity<OrderResponse> createOrder(
            @RequestBody @Valid CreateOrderRequest request) {
        // HTTP Request DTO → Service DTO 변환 (Presentation 책임)
        OrderDto orderDto = orderService.createOrder(request.toServiceDto());
        // Service DTO → HTTP Response DTO 변환 (Presentation 책임)
        return ResponseEntity.ok(OrderResponse.from(orderDto));
    }

    @GetMapping("/{id}")
    public ResponseEntity<OrderResponse> getOrder(@PathVariable Long id) {
        OrderDto orderDto = orderService.findById(id);
        return ResponseEntity.ok(OrderResponse.from(orderDto));
    }

    // 예외 처리도 Presentation 계층 책임
    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleOrderNotFound(OrderNotFoundException e) {
        return ResponseEntity.status(404).body(new ErrorResponse(e.getMessage()));
    }
}
```

책임:
- HTTP 요청/응답 처리 (상태 코드, 헤더)
- 입력 유효성 검사 (`@Valid`)
- DTO 변환 (Request → ServiceDto, ServiceDto → Response)
- 인증/인가 처리 (Spring Security 필터와 연계)

### Business Layer (비즈니스 계층)

**핵심 비즈니스 로직**을 담당합니다. 트랜잭션 경계도 이 계층에서 관리합니다.

```java
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)  // 기본 읽기 전용 트랜잭션
public class OrderService {

    private final OrderRepository orderRepository;
    private final ProductRepository productRepository;
    private final InventoryService inventoryService;
    private final NotificationService notificationService;

    @Transactional  // 쓰기 작업은 별도 트랜잭션
    public OrderDto createOrder(CreateOrderServiceDto dto) {
        // 1. 비즈니스 규칙 검증 (재고 확인)
        dto.items().forEach(item ->
            inventoryService.validateStock(item.productId(), item.quantity())
        );

        // 2. 도메인 로직 (주문 생성)
        Order order = Order.create(dto.customerId(), buildOrderItems(dto.items()));
        Order saved = orderRepository.save(order);

        // 3. 부수 효과 처리 (재고 차감, 알림)
        inventoryService.deductStock(saved);
        notificationService.sendOrderConfirmation(saved);

        return OrderDto.from(saved);
    }

    public OrderDto findById(Long id) {
        return orderRepository.findById(id)
            .map(OrderDto::from)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }

    private List<OrderItem> buildOrderItems(List<OrderItemDto> itemDtos) {
        return itemDtos.stream()
            .map(item -> {
                Product product = productRepository.findById(item.productId())
                    .orElseThrow(() -> new ProductNotFoundException(item.productId()));
                return new OrderItem(product, item.quantity());
            })
            .toList();
    }
}
```

### Persistence Layer (영속 계층)

**데이터 저장/조회**를 담당합니다.

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    // 단순 조회
    List<Order> findByCustomerId(Long customerId);

    // JPQL 복잡 조회 (N+1 방지를 위한 fetch join)
    @Query("""
        SELECT o FROM Order o
        JOIN FETCH o.items
        WHERE o.status = :status
        AND o.createdAt >= :from
        """)
    List<Order> findByStatusAndCreatedAtAfter(
        @Param("status") OrderStatus status,
        @Param("from") LocalDateTime from
    );

    // 통계 쿼리 (Native SQL)
    @Query(value = """
        SELECT DATE(created_at) as date,
               COUNT(*) as count,
               SUM(total_amount) as revenue
        FROM orders
        WHERE created_at BETWEEN :from AND :to
        GROUP BY DATE(created_at)
        """, nativeQuery = true)
    List<DailySummaryProjection> findDailySummary(
        @Param("from") LocalDateTime from,
        @Param("to") LocalDateTime to
    );
}
```

#### Application Layer 예시 (Facade)

```java
// 여러 서비스를 조합하는 오케스트레이션 역할
@Service
@RequiredArgsConstructor
public class CheckoutFacade {

    private final CartService cartService;
    private final OrderService orderService;
    private final PaymentService paymentService;
    private final CouponService couponService;

    @Transactional
    public CheckoutResult checkout(CheckoutRequest request) {
        // 1. 장바구니 검증
        Cart cart = cartService.getCart(request.cartId());

        // 2. 쿠폰 적용
        Money discountedAmount = couponService.apply(cart.getTotalAmount(), request.couponCode());

        // 3. 주문 생성
        Order order = orderService.createFromCart(cart, discountedAmount);

        // 4. 결제 처리
        PaymentResult payment = paymentService.charge(order, request.paymentMethod());

        return CheckoutResult.of(order, payment);
    }
}
```

## 트래픽 시나리오별 분석

### 트래픽 적을 때 (100 TPS)

```
이 시점에서 레이어드 아키텍처가 최적:
- 단순한 구조로 빠른 개발 가능
- 모든 요청이 Controller → Service → Repository → DB 순서로 처리
- DB 쿼리 최적화가 불필요한 수준

평균 응답 시간: 20~50ms (DB 쿼리 포함)
병목: 없음
```

### 트래픽 높을 때 (10,000 TPS)

```
레이어드 아키텍처에서 발생하는 문제:
1. Service 레이어가 비대해짐 → 단일 클래스 수천 줄
2. 동일한 데이터를 여러 Service가 반복 조회 → DB 부하
3. 캐싱 로직이 Service 코드에 침투 → 비즈니스 로직 오염

해결 전략:
- Application Layer(Facade) 추가로 오케스트레이션 분리
- Repository에 캐싱 적용 (Service 코드 변경 없음)
- 읽기 전용 Repository 분리 (@Transactional(readOnly=true))

예시:
@Service
public class OrderQueryService {  // 조회 전용 Service 분리
    @Transactional(readOnly = true)
    public OrderDto findById(Long id) { ... }
}

@Service
public class OrderCommandService {  // 수정 전용 Service 분리
    @Transactional
    public OrderDto createOrder(...) { ... }
}
```

### 극한 시나리오 (100,000+ TPS)

```
레이어드 아키텍처의 한계 도달:
- Service 레이어 자체가 병목 (단일 JVM)
- 수평 확장 시 레이어별 독립 스케일링 불가
- DB 연결 풀 고갈 (모든 레이어가 동일 DB)

이 시점에서 전환 필요:
1. CQRS: Command/Query 레이어 분리
   → 조회: ElasticSearch, Redis (빠른 읽기)
   → 수정: MySQL (트랜잭션 보장)

2. MSA: 레이어 → 독립 서비스로 분리
   → 각 서비스가 독립적으로 스케일 가능

3. 이벤트 기반 비동기 처리
   → Kafka로 레이어 간 통신 → 응답 시간 단축

레이어드가 MSA의 기초이므로, 레이어드를 잘 설계해야
나중에 서비스 분리가 쉽다.
```

## 장단점 분석

### 장점

1. **이해하기 쉬움**: 새 팀원 온보딩이 빠름, 업계 표준 패턴
2. **역할 분리**: Controller는 HTTP, Service는 비즈니스, Repository는 DB
3. **검증된 패턴**: 수십 년간 실무에서 검증됨
4. **Spring 기본 지원**: Spring MVC, Spring Data JPA가 이 패턴을 기본으로 지원
5. **점진적 개선 가능**: 필요할 때 헥사고날이나 DDD로 전환 용이

### 단점

1. **DB 중심 설계로 흐르기 쉬움**: DB 테이블 구조가 그대로 Entity → Service → Controller로 올라옴
2. **단위 테스트 어려움**: Service 테스트 시 Repository Mock 필요
3. **레이어 스킵 유혹**: Controller → Repository 직접 호출, 비즈니스 로직이 Controller로 새어나옴
4. **순환 의존성 위험**: Service A → Service B → Service A
5. **확장성 한계**: 모놀리식 구조, 특정 레이어만 독립 스케일 불가

### 뚱뚱한 Controller (Fat Controller)

```java
// 나쁜 예 - 비즈니스 로직이 Controller에 있음
@PostMapping("/order")
public ResponseEntity<?> createOrder(@RequestBody OrderRequest request) {
    // 비즈니스 로직이 컨트롤러에 - Controller 책임 초과
    if (request.getQuantity() <= 0) throw new BadRequestException("수량 오류");
```

### 빈약한 도메인 모델 (Anemic Domain Model)

```java
// 나쁜 예 - Entity에 로직이 없고 Service에 모든 로직이 집중
public class Order {
    private Long id;
    private String status; // getter/setter만 존재
    public String getStatus() { return status; }
    public void setStatus(String status) { this.status = status; }
}

public class OrderService {
    public void cancelOrder(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        // 비즈니스 로직이 서비스에 있음 - Entity가 단순 데이터 홀더
        if (order.getStatus().equals("SHIPPED")) throw new Exception("배송 중");
        order.setStatus("CANCELLED");
    }
}

// 좋은 예 - 비즈니스 로직이 Entity 안에
public class Order {
    private OrderStatus status;

    // "주문을 취소한다"는 비즈니스 행위가 Entity에
    public void cancel() {
        if (status == OrderStatus.SHIPPED) {
            throw new OrderAlreadyShippedException("배송 중인 주문은 취소 불가");
        }
        this.status = OrderStatus.CANCELLED;
    }
}
```

### 순환 의존성

```java

@Service
public class OrderService {

}

@Service
public class PaymentService {

// 해결 방법:
// 1. 공통 로직을 별도 서비스로 추출
```

## 실무에서 자주 하는 실수

#### 실수 1: @Transactional 위치 오류

```java
// 나쁜 예 - readOnly=true 트랜잭션에서 쓰기 시도
@Transactional(readOnly = true)  // 클래스 레벨
public class OrderService {

    public void createOrder(CreateOrderDto dto) {  // readOnly 상속!
        Order order = new Order(dto);
        orderRepository.save(order);  // 실패! readOnly 트랜잭션에서 쓰기
    }
}

// 좋은 예 - 메서드 레벨로 오버라이드
@Transactional(readOnly = true)  // 기본: 읽기 전용
public class OrderService {

    @Transactional  // 쓰기 메서드는 별도 지정
    public void createOrder(CreateOrderDto dto) {
        orderRepository.save(new Order(dto));
    }

    public OrderDto findById(Long id) {  // readOnly 상속
        return orderRepository.findById(id).map(OrderDto::from).orElseThrow();
    }
}
```

#### 실수 2: DTO를 계층 간 잘못 공유

```java
// 나쁜 예 - HTTP Request DTO가 Service까지 내려감
@Service
public class OrderService {
    public void createOrder(CreateOrderRequest request) {  // HTTP DTO 사용
        // Service가 HTTP 레이어를 알게 됨 → 레이어 위반
        // Request에 @JsonProperty 등 HTTP 관련 어노테이션이 있으면 더 심각
    }
}

// 좋은 예 - 계층 경계에서 DTO 변환
@RestController
public class OrderController {
    public ResponseEntity<?> createOrder(@RequestBody CreateOrderRequest request) {
        CreateOrderServiceDto dto = request.toServiceDto();  // Controller에서 변환
        orderService.createOrder(dto);  // Service는 자체 DTO만 사용
    }
}
```

## 면접 포인트

#### Q. 레이어드 아키텍처에서 각 계층의 의존성 방향은?

```
단방향: Presentation → Business → Persistence
역방향 금지: Persistence가 Business를 알면 안 됨

이유:
- 상위 계층(Presentation)이 하위 계층(Business)을 호출
- 하위 계층은 상위 계층을 모름
- Business 로직이 HTTP나 DB 기술에 의존하지 않음
```

#### Q. 레이어드 vs 헥사고날 선택 기준은?

```
레이어드 선택:
- 팀 규모 작음 (5인 이하)
- CRUD 위주, 비즈니스 로직 단순
- 빠른 MVP 개발 필요
- 외부 시스템 교체 가능성 낮음

헥사고날 선택:
- 장기 프로젝트, 큰 팀
- 외부 시스템 교체 가능성 높음
- 단위 테스트 커버리지가 중요
- DDD를 함께 적용할 때
```

#### Q. Anemic Domain Model이 왜 문제인가?

```
빈약한 도메인 모델 = Entity에 비즈니스 로직 없음
→ 모든 로직이 Service에 집중
→ Service가 수천 줄짜리 거대 클래스가 됨
→ 비즈니스 규칙이 여러 Service에 중복 존재
→ Entity는 단순 DTO와 구별 불가

해결: Entity에 비즈니스 행위를 담는다
  order.cancel()  // Entity가 스스로 취소 가능한지 판단
  order.addItem() // Entity가 항목 추가 규칙을 앎
```

## 헥사고날 아키텍처와 비교

| 항목 | 레이어드 | 헥사고날 |
|------|---------|---------|
| 진입 장벽 | 낮음 | 높음 |
| 인프라 교체 용이성 | 낮음 | 높음 |
| 단위 테스트 | 보통 (Mock 필요) | 쉬움 (Port Mock) |
| 코드량 | 적음 | 많음 |
| 소규모/단기 | 최적 | 과도 |
| 대규모/장기 | 유지보수 어려움 | 적합 |
| DB 교체 | 전면 수정 | Adapter만 교체 |
| 외부 API 교체 | Service 수정 | Adapter만 교체 |

## 핵심 포인트 정리

```
1. 3계층 기본 원칙
   Presentation(HTTP) → Business(비즈니스) → Persistence(DB)
   의존성은 위에서 아래로만 흐름 (역방향 금지)

2. 실무에서 4계층 추천
   Presentation → Application(Facade) → Domain → Infrastructure
   Application Layer로 복잡한 오케스트레이션 분리

3. 안티패턴 3가지
   ① Fat Controller: 비즈니스 로직이 Controller에
   ② Layer Skip: Controller → Repository 직접 접근
   ③ Anemic Model: Entity에 로직 없음, Service에 집중

4. DTO 변환은 계층 경계에서
   Controller: HTTP Request DTO → Service DTO 변환
   Service: Domain Entity → Service DTO 변환

5. 트랜잭션 관리는 Service 계층에서
   @Transactional(readOnly=true) 기본
   쓰기 메서드에 @Transactional 별도 지정

6. 언제 전환할까?
   레이어드 → 헥사고날: 외부 시스템 교체 빈번, 테스트 중요해질 때
   레이어드 → MSA: 특정 도메인 독립 배포/스케일이 필요할 때
```
