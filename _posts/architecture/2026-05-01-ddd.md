> **한 줄 요약**: DDD는 비즈니스 도메인의 복잡성을 소프트웨어 구조에 직접 반영하는 설계 방법론으로, "기술이 아닌 도메인이 설계의 중심"이다.

## 비유로 시작하기

대형 병원을 떠올려 보세요. 내과, 외과, 약제부, 원무과가 모두 "환자"를 다루지만, 각 부서가 바라보는 "환자"의 의미는 완전히 다릅니다.

- 원무과의 환자: 결제 정보, 보험 번호가 핵심
- 내과의 환자: 진단 이력, 복용 약물이 핵심
- 약제부의 환자: 처방전 목록, 알레르기 정보가 핵심

같은 단어(환자)지만 부서마다 다른 의미를 가집니다. 이걸 소프트웨어에서도 명확히 분리하자는 게 DDD의 출발점입니다.

DDD(Domain-Driven Design)는 **비즈니스 도메인의 복잡성을 소프트웨어에 정확하게 반영**하기 위한 설계 방법론입니다. 에릭 에반스(Eric Evans)가 2003년 저서에서 정립했으며, "기술이 아닌 도메인이 설계의 중심"이라는 철학을 가집니다.

---

## 왜 DDD가 중요한가?

```
전통적 접근 (기술 중심):
  개발자: "users 테이블과 orders 테이블을 JOIN해서 LEFT OUTER로..."
  기획자: "그래서 고객이 주문을 취소할 수 있다는 거예요?"

DDD 접근 (도메인 중심):
  개발자: "고객이 주문을 취소하면 OrderCancelledEvent가 발행됩니다"
  기획자: "네! 그러면 배송팀이 배송을 중단하고, 결제팀이 환불 처리를 하겠군요"
```

기술 언어가 아닌 비즈니스 언어로 소통할 때:
- 요구사항 오해가 줄어듦
- 코드가 비즈니스 규칙을 직접 표현
- 도메인 전문가가 코드를 읽고 검증 가능

## 전략적 설계

### Ubiquitous Language (유비쿼터스 언어)

개발자와 도메인 전문가가 **동일한 용어**를 사용하는 것입니다. "주문"이 개발자에게는 `Order` 테이블이고, 영업팀에게는 "계약서"라면 의사소통 비용이 폭발합니다.

```
나쁜 예 (기술 언어):
  개발자: "user_product_mapping 테이블에 insert 했어요"
  기획자: "?????"

좋은 예 (유비쿼터스 언어 적용):
  개발자: "고객이 상품을 장바구니에 담았습니다"
  기획자: "네, 그 후 결제하면 주문이 생성되는군요"
```

유비쿼터스 언어는 회의록, 코드, 문서, 테스트 이름까지 모두 동일하게 사용해야 합니다.

```java
// 나쁜 코드 - 기술 용어 남발
userProductMappingRepository.insertMapping(userId, productId, qty);

// 좋은 코드 - 유비쿼터스 언어 반영
cart.addItem(product, quantity);  // "장바구니에 상품을 담는다"
```

### Entity (엔티티)

**고유 식별자(Identity)를 가지며, 생명주기 동안 상태가 변하는 객체**입니다.

        STATE["상태 변경\n가능"]
        EQUAL["ID가 같으면\n동일 객체"]
    end

    ID --- STATE
    STATE --- EQUAL

    class ID,STATE,EQUAL feature

```java
@Entity
public class Order {
    @Id
    private OrderId id;          // 식별자 - 이게 있어야 Entity
    private CustomerId customerId;
    private OrderStatus status;
    private List<OrderItem> items;
    private Money totalAmount;

    // 도메인 로직이 엔티티 안에 있어야 한다
    public void cancel() {
        if (this.status == OrderStatus.SHIPPED) {
            // 비즈니스 규칙이 코드로 표현됨
            throw new OrderAlreadyShippedException("배송 중인 주문은 취소할 수 없습니다");
        }
        this.status = OrderStatus.CANCELLED;
        // 도메인 이벤트 발행 - "주문이 취소되었다"는 사실 기록
        registerEvent(new OrderCancelledEvent(this.id));
    }

    public void addItem(Product product, int quantity) {
        validateOrderStatus();  // "주문 가능 상태인가?" - 비즈니스 규칙
        OrderItem item = new OrderItem(product.getId(), product.getPrice(), quantity);
        this.items.add(item);
        this.totalAmount = this.totalAmount.add(item.getSubtotal());
    }
}
```

핵심: **식별자가 같으면 동일 객체**. 속성이 달라도 id가 같으면 같은 엔티티입니다.

### Value Object (값 객체)

**식별자가 없고, 속성의 조합으로 동일성을 판단**하는 불변 객체입니다.

```java
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money(BigDecimal amount, Currency currency) {
        // 생성 시점에 유효성 검증 - 항상 유효한 상태
        if (amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("금액은 음수일 수 없습니다");
        }
        this.amount = amount;
        this.currency = currency;
    }

    // 불변 객체: 연산 결과는 새 객체 반환 (기존 객체 변경 없음)
    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new CurrencyMismatchException("통화가 다릅니다: " + this.currency + " vs " + other.currency);
        }
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int multiplier) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(multiplier)), this.currency);
    }

    // 동일성 = 값의 조합 (id 없음)
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Money)) return false;
        Money m = (Money) o;
        return this.amount.equals(m.amount) && this.currency.equals(m.currency);
    }
}
```

좋은 VO 후보: `Money`, `Address`, `Email`, `PhoneNumber`, `DateRange`, `Coordinate`, `OrderStatus`, `ProductId`

#### VO vs Entity 판단 기준

| 질문 | VO | Entity |
|------|-----|--------|
| 고유 식별자가 필요한가? | 아니오 | 예 |
| 독립적 생명주기가 있는가? | 아니오 | 예 |
| 값이 같으면 같은 것인가? | 예 | 아니오 (id 기준) |
| 변경이 필요한가? | 새 객체 생성 | 상태 변경 |
| 예시 | Money(10000원), Address | Order, User |

### Aggregate (집합체)

Aggregate 설계 원칙:
1. **하나의 트랜잭션 = 하나의 Aggregate 수정**
2. **다른 Aggregate 참조는 ID로만**
3. **Aggregate 크기는 작게 유지**

```java
// 나쁜 예 - Aggregate 경계 위반
orderItem.setPrice(newPrice);  // Root를 거치지 않고 직접 수정

// 좋은 예 - Root를 통해 수정
order.updateItemPrice(orderItemId, newPrice);  // 비즈니스 규칙 검증 포함

// 나쁜 설계 - Aggregate가 너무 큼
class Order {
    private Customer customer;       // 전체 객체 포함 → 락 범위 너무 넓음
    private List<Product> products;  // 전체 객체 포함 → 동시성 병목
}

// 좋은 설계 - ID로만 참조
class Order {
    private CustomerId customerId;    // ID만 참조 → 락 범위 최소화
    private List<OrderItem> items;    // items에 productId, 당시 가격만 보관
}
```

### Repository (리포지토리)

**Aggregate의 영속성을 추상화**하는 컬렉션처럼 동작하는 인터페이스입니다.

```java
// 도메인 계층의 인터페이스 (인프라 무관) - "무엇을"만 정의
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
    List<Order> findByCustomerId(CustomerId customerId);
    void delete(OrderId id);
    // DB, Redis, 파일 등 구현 방식은 모름 - 알 필요 없음
}

// 인프라 계층의 구현체 - "어떻게"를 정의
@Repository
public class JpaOrderRepository implements OrderRepository {
    private final OrderJpaRepository jpaRepo;
    private final OrderMapper mapper;

    @Override
    public Order findById(OrderId id) {
        return jpaRepo.findById(id.getValue())
            .map(mapper::toDomain)          // JPA Entity → Domain Entity 변환
            .orElseThrow(() -> new OrderNotFoundException(id));
    }

    @Override
    public void save(Order order) {
        OrderEntity entity = mapper.toEntity(order);  // Domain → JPA Entity 변환
        jpaRepo.save(entity);
    }
}
```

### Domain Event (도메인 이벤트)

**도메인에서 발생한 중요한 사건**을 나타냅니다. Aggregate 간 통신 및 부수 효과를 이벤트 기반으로 처리할 때 사용합니다.

```java
// 이벤트 정의 - 과거형, 불변 객체
public record OrderPlacedEvent(
    String eventId,           // 멱등성 처리용
    OrderId orderId,
    CustomerId customerId,
    Money totalAmount,
    Instant occurredAt        // 발생 시각
) {
    public OrderPlacedEvent(Order order) {
        this(UUID.randomUUID().toString(), order.getId(),
             order.getCustomerId(), order.getTotalAmount(), Instant.now());
    }
}

// Aggregate에서 이벤트 발행
public class Order extends AbstractAggregateRoot<Order> {
    public void place() {
        this.status = OrderStatus.PLACED;
        // "주문이 접수되었다"는 사실을 이벤트로 기록
        registerEvent(new OrderPlacedEvent(this));
    }
}

// 이벤트 핸들러 - 다른 Bounded Context에서 반응
@Component
public class NotificationEventHandler {
    @EventListener
    public void handleOrderPlaced(OrderPlacedEvent event) {
        // Order BC는 Notification BC를 모름 - 느슨한 결합
        notificationService.sendOrderConfirmation(
            event.customerId(), event.orderId());
    }
}
```

### Domain Service (도메인 서비스)

**단일 Entity나 VO에 속하기 애매한 도메인 로직**을 담습니다. 상태를 갖지 않습니다.

```java
// 할인 정책 계산 - Order, Customer, Coupon 세 Aggregate가 모두 관여
// → 어느 하나에 넣기 어려움 → Domain Service
@DomainService
public class DiscountCalculationService {

    // 도메인 전문가와 함께 정의한 규칙을 코드로 표현
    public Money calculateDiscount(Order order, Customer customer, List<Coupon> coupons) {
        // 구매 금액 기준 볼륨 할인
        Money baseDiscount = calculateVolumeDiscount(order);
        // 회원 등급 할인 (VIP, 골드, 실버)
        Money memberDiscount = calculateMemberDiscount(customer);
        // 쿠폰 할인 (중복 적용 가능)
        Money couponDiscount = coupons.stream()
            .map(c -> c.apply(order.getTotalAmount()))
            .reduce(Money.ZERO, Money::add);
        return baseDiscount.add(memberDiscount).add(couponDiscount);
    }
}
```

중요한 점: **도메인 계층은 다른 계층에 의존하지 않습니다**. 인프라 의존성은 인터페이스(Repository)로 역전시킵니다.

## 트래픽 시나리오별 DDD 적용

### 트래픽 적을 때 (100 TPS)

```
상황: 스타트업 초기
- Aggregate 경계가 다소 느슨해도 문제없음
- 단일 DB 트랜잭션으로 여러 Aggregate 수정 가능
- 성능보다 비즈니스 로직 명확성이 우선

권장: DDD 전술 패턴 적용하되, 인프라 단순화
```

### 트래픽 높을 때 (10,000 TPS)

```
상황: 성장기 서비스
- Aggregate 크기가 성능에 직접 영향
- 큰 Aggregate = 락 범위 넓음 = 동시성 병목

예시: Order Aggregate에 고객 정보까지 포함 시
  → 주문 1건 처리할 때마다 고객 레코드 락
  → 같은 고객의 동시 주문 처리 불가
  → TPS 급락

해결: Aggregate 최소화, 다른 BC 참조는 ID로만
```

### 극한 시나리오 (100,000+ TPS)

```
상황: 대규모 트래픽 (쇼핑몰 타임딜)

문제: Order Aggregate에 items가 100개인 경우
  → findById 시 items 100개 모두 로딩
  → 1초에 수천 건 조회 → DB 부하 폭발

해결:
1. Aggregate 분리: OrderSummary(Root) + OrderItems(별도 Aggregate)
2. 이벤트 소싱: 이벤트를 재생해 상태 복원 (DB 읽기 최소화)
3. CQRS: 조회 전용 ReadModel 분리 (Aggregate 로딩 없이 조회)

```java
// CQRS Read Model - Aggregate 로딩 없이 빠른 조회
@Query("SELECT new OrderSummaryDto(o.id, o.status, o.totalAmount) FROM Order o WHERE o.customerId = :customerId")
List<OrderSummaryDto> findSummariesByCustomerId(@Param("customerId") Long customerId);
```
```

## 실무 적용 사례

### 이커머스 주문 시스템 Bounded Context 분리

```
Bounded Context 분리:
├── 상품 컨텍스트 (Product BC)
│   - 상품 등록/수정, 카탈로그 관리
│   - Aggregate: Product, Category
├── 주문 컨텍스트 (Order BC)
│   - 장바구니, 주문 생성, 결제
│   - Aggregate: Order, Cart
├── 배송 컨텍스트 (Delivery BC)
│   - 배송 처리, 추적
│   - Aggregate: Shipment, DeliveryRoute
├── 정산 컨텍스트 (Settlement BC)
│   - 판매자 정산, 수수료
│   - Aggregate: Settlement, Commission
└── 알림 컨텍스트 (Notification BC)
    - 이메일, 푸시, SMS
    - Aggregate: Notification, Template
```

각 BC 간 통신은 도메인 이벤트(Kafka)로 비동기 처리하여 결합도를 낮춥니다.

## 실무에서 자주 하는 실수

#### 실수 1: Aggregate가 너무 큰 경우

```java
// 나쁜 설계 - 모든 것을 Order에 포함
class Order {
    private Customer customer;        // Customer Entity 전체 포함 → 락 범위 폭발
    private List<Product> products;   // Product Entity 전체 포함 → N+1 문제
    private Inventory inventory;      // Inventory까지 → 동시 주문 불가
}

// 해결 - ID만 참조
class Order {
    private CustomerId customerId;        // ID 참조만
    private List<OrderItem> items;        // OrderItem에 당시 가격 스냅샷 보관
    // Inventory는 별개 Aggregate, 이벤트로 통신
}
```

#### 실수 2: 빈약한 도메인 모델 (Anemic Domain Model)

```java
// 나쁜 설계 - Entity에 로직이 없고 Service에 모든 로직
class Order {
    private String status;
    // getter/setter만 존재 - 데이터 클래스
}

class OrderService {
    public void cancelOrder(Long id) {
        Order order = repo.findById(id).orElseThrow();
        if (order.getStatus().equals("SHIPPED")) throw ...;
        order.setStatus("CANCELLED");  // 비즈니스 로직이 서비스에 있음
    }
}

// 좋은 설계 - 도메인 로직이 Entity 안에
class Order {
    public void cancel() {  // "주문을 취소한다"는 비즈니스 행위
        if (status == SHIPPED) throw new OrderAlreadyShippedException();
        this.status = CANCELLED;
        registerEvent(new OrderCancelledEvent(id));
    }
}
```

#### 실수 3: Bounded Context 경계 혼재

```
나쁜 패턴: Order Service가 Delivery DB에 직접 쿼리
  SELECT * FROM delivery.shipments WHERE order_id = ?
  → 두 BC가 DB 수준에서 결합 → 독립 배포 불가

올바른 패턴:
  Order Service → OrderShippedEvent → Delivery Service
  또는 Delivery Service REST API 호출 (ACL로 격리)
```

## 면접 포인트

#### Q. Entity와 Value Object의 차이는?

```
Entity: 고유 식별자(ID)가 있고, 생명주기 동안 상태가 변함.
        같은 속성이라도 ID가 다르면 다른 객체.
        예) Order(주문), User(사용자)

Value Object: 식별자 없음, 속성 조합으로 동일성 판단, 불변.
              예) Money(금액+통화), Address(주소), Email
              new Money(1000, KRW) == new Money(1000, KRW) → true
```

#### Q. Aggregate 설계 시 가장 중요한 원칙은?

```
1. 트랜잭션 경계 = Aggregate 경계
   (하나의 트랜잭션에서 하나의 Aggregate만 수정)

2. 다른 Aggregate 참조는 반드시 ID로만
   (전체 객체 참조 시 락 범위 증가, 의존성 증가)

3. Aggregate는 가능한 작게
   (크기 = 락 범위 = 성능 저하)
```

#### Q. Domain Event는 언제 사용하나?

```
사용 시점:
- Aggregate 간 통신이 필요할 때
- 한 BC의 변경이 다른 BC에 영향을 줄 때
- 비동기 처리가 적합한 부수 효과 (알림, 통계 등)

예) Order.place() → OrderPlacedEvent → Payment 처리 시작
   (Order BC는 Payment BC를 직접 호출하지 않음)
```

## 핵심 포인트 정리

```
1. DDD = 비즈니스 언어로 코드 작성하는 방법론
   → 도메인 전문가 + 개발자가 같은 언어(Ubiquitous Language) 사용

2. Bounded Context = 동일 언어가 동일 의미를 갖는 경계
   → 이상적으로 마이크로서비스 1개 = Bounded Context 1개

3. Aggregate = 트랜잭션 일관성 단위
   → 하나의 트랜잭션 = 하나의 Aggregate 수정
   → 가능한 작게, 다른 Aggregate는 ID로만 참조

4. Entity vs Value Object
   → Entity: ID 있음, 상태 변경 가능
   → VO: ID 없음, 불변, 값으로 동일성 판단

5. Domain Event = Aggregate 간 느슨한 결합
   → 직접 호출 대신 이벤트 발행으로 BC 간 통신
```
